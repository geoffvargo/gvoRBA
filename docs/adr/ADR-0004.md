# 4. Scope the room bookings endpoint to a date range, not a single date

- **Status:** Accepted (supersedes the single-date endpoint defined in the v0.1 draft)
- **Date:** 2026-07-09
- **Deciders:** Geoff Vargo

## Context

The v0.1 spec contradicted itself: `GET /api/rooms/{id}/bookings?date=YYYY-MM-DD` returns one day,
while FR-2.3 called for a calendar spanning two weeks. Bounding the endpoint was right; bounding it
to *one day* was arbitrary, and it constrained the frontend — rendering a week would have cost five
sequential requests. The room page exists to answer "when can I book this room?", and under FR-3.1's
30-day horizon a day view makes that up to 30 navigations. A Mon–Fri grid answers it at a glance.

## Decision

`GET /api/rooms/{id}/bookings` accepts an inclusive date range: `?from=YYYY-MM-DD&to=YYYY-MM-DD`.

- `to` is optional and defaults to `from`, preserving a cheap single-day fetch for the
  409-conflict refresh. No calendar view requests a single day.
- The range may not exceed 31 days; longer ranges are rejected with `VALIDATION_FAILED`. This
  preserves the bounded-payload property that motivated the original `?date=` param. FR-3.1's
  30-day horizon means nothing beyond that window is bookable anyway.
- The calendar on `/rooms/:id` renders whole business weeks (Mon–Fri). The span is user-selectable
  from a minimum of one business week to a maximum of four — "one business month". Four business
  weeks spans at most 26 calendar days, inside the 31-day cap above. The calendar never renders a
  span shorter than one business week, on any viewport.
- The 409-conflict refresh (§8) issues a single-day request, since only the affected day changes.

## Alternatives considered

- **Unbounded `GET /{id}/bookings`, filter client-side.** Works at portfolio scale and would be
  measurably no slower. Rejected because FR-3.7 retains cancelled bookings indefinitely, so the
  response grows monotonically and the endpoint needs a bound eventually. Better to have one now
  than to retrofit pagination later.
- **`?weekOf=YYYY-MM-DD`.** Encodes the current UI into the API contract, and cannot express a
  multi-week span or the single-day 409 refresh. A range param serves all three.

## Consequences

- **Positive:** One endpoint serves every span the product needs — one day for the 409 refresh,
  one to four business weeks for the calendar. The API is scoped to a bounded window without being
  scoped to a specific UI, so the rendered span can change without a backend change.
- **Negative:** The controller must validate `from <= to` and enforce the 31-day cap, which is
  logic the single-date endpoint did not need. Two query params instead of one is marginally more
  surface for OpenAPI docs and integration tests to cover.
- **Follow-up:** The caller chain is new work, not a refactor — no method on `master` calls a
  range query today.
  - `BookingRepository.findForRoomInRange(roomId, startInclusive, endExclusive)` — new. Its overlap
    predicate is identical to the single-date form; only its bound arguments widen.
  - `RoomController.getRoomBookings(id, from, to)` — new. `RoomController` must inject
    `BookingRepository`, which it currently does not. Validates `from <= to` and the 31-day cap,
    and defaults `to` to `from` when absent.
  - `ApiService.getRoomBookings(id, from, to)` — new. Replaces nothing; the existing
    `ApiService.getBookings()` is unrelated and stays.
  - `RoomStore.loadRoomBookings(roomId, from, to)` — new. `RoomStore` keeps `_selectedDate` as the
    anchor signal and adds `_spanWeeks` (1–4, default 1); `weekStart()` and `rangeEnd()` are
    `computed()` from the two, so span selection stays a rendering concern rather than a state
    concern.
