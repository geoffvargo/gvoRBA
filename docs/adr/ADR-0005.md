# 5. Calendar Visualization for a Room's Bookings for the Week

- **Status:** Accepted
- **Date:** 2026-07-14
- **Deciders:** Geoff Vargo

## Context

ADR-0004 settled the calendar's data contract but left rendering open. `/rooms/:id` must overlay a
variable set of bookings onto a fixed grid: weekdays across, hours down, each block positioned by
start time and sized by duration.

A third-party calendar was never on the table. It would reduce the project's most visually prominent
feature to a configuration exercise, and would introduce RxJS-oriented APIs into a codebase that is
signal-based per ADR-0003. The remaining question was only how to express the layout.

## Decision

Build the calendar as CSS Grid + subgrid driven by Angular style bindings.

- `#calendar-canvas` declares the tracks once: `grid-cols-[repeat(6,8rem)]` (hour gutter + five
  weekdays), `grid-rows-[2rem,repeat(10,4rem)]` (header + 8am–5pm).
- `#hours` and `#schedule-grid` take `grid-cols-subgrid grid-rows-subgrid`, so labels, hashmarks,
  and bookings resolve against those same tracks. Rescaling the canvas rescales every layer.
- `#schedule-grid` subdivides into `grid-rows-[repeat(40,1rem)]` — 15 minutes is the quantum.
- Blocks are placed by `[style.grid-column]="dayMapper(booking.startsAt)"` and
  `[style.grid-row-start|-end]="timeMapper(...)"`. Duration is a grid span computed from `endsAt`,
  so the browser does the arithmetic — no pixel scale hardcoded in TypeScript, and the layout
  rescales through CSS alone.
## Consequences

- **Positive:** Zero calendar dependencies. Time scale declared once, inherited via subgrid. Blocks
  are ordinary DOM nodes — themeable, focusable, and available as event targets, which is what
  click-to-book and drag interactions will build on. Positioning is a pure function of
  `startsAt`/`endsAt` — unit-testable without a DOM.
- **Negative:** Overlapping bookings stack rather than split the column (latent only, given
  ADR-0001's exclusion constraint). Subgrid needs Chrome 117+/Safari 16+.
- **Known defects (fix before Milestone 3 closes):**
    1. `dayMapper` returns `getDay()` (Mon = 1), but Monday is `col-start-2` — every block renders one
       column early. Needs `getDay() + 1`.
    2. `rowGridMapper` uses `(hours - 7)` against a grid starting at 8:00, and subtracts 1 from a
       1-based grid line. 8:00 resolves to line 3.
    3. Template calls `timeMapper()`/`dayMapper()` on every CD pass; per ADR-0003 these belong in a
       `computed()` view-model.
- **Follow-up:** The sub-`md` day view (ADR-0004) reuses this grid; only the column mapper changes.
  Interactivity (click-to-book, drag-to-create) attaches to the existing block and cell elements
  without changing the layout model.
