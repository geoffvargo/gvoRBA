# 2. Use stateless JWT authentication

- **Status:** Accepted
- **Date:** 2026-06-09
- **Deciders:** Geoff Vargo

## Context

The frontend is an Angular single-page application talking to a Spring Boot
REST API. We need an authentication mechanism that supports role-based access
(guest / member / admin) and allows the API to be scaled horizontally across
multiple stateless instances without requiring sticky sessions or a shared
session store.

## Decision

We will use **stateless authentication with JSON Web Tokens**:

- A short-lived **access token** (~15 min) carries the user id and roles and is
  validated on each request by Spring Security configured as an OAuth2 resource
  server.
- A longer-lived **refresh token** is issued to obtain new access tokens
  without re-entering credentials.
- The **access token is held in memory** by the Angular app and attached via an
  HTTP interceptor; the **refresh token is stored in an `HttpOnly`, `Secure`,
  `SameSite` cookie** so it is not reachable by JavaScript.

Authorization decisions use the roles claim via method-level security
(`@PreAuthorize`) and Angular route guards mirror the same rules client-side.

## Alternatives considered

- **Server-side sessions.** Simpler, and revocation is instant. Rejected as the
  default because it requires either sticky sessions or a shared session store
  (e.g. Redis), adding infrastructure and coupling that this project does not
  need.
- **OAuth2 / OIDC via an external provider (Auth0, Keycloak, Google).** The
  right choice for a real product and worth mentioning in interviews, but it
  adds setup overhead and obscures the auth fundamentals this project is meant
  to demonstrate.

## Consequences

- **Positive:** The API is fully stateless and scales horizontally with no
  shared session state. The split token storage balances XSS risk (access token
  not persisted) against CSRF risk (refresh token in an `HttpOnly` cookie).
- **Negative:** Token revocation is not instantaneous — a leaked access token is
  valid until it expires. We mitigate with short access-token lifetimes and
  refresh-token rotation; a denylist could be added later if needed.
- **Follow-up:** Ensure clock skew tolerance is configured and that the refresh
  endpoint is protected against CSRF.
