# 1. Use optimistic locking to prevent double-bookings

- **Status:** Accepted
- **Date:** 2026-03-14
- **Deciders:** Geoff Vargo

## Context

The core invariant of the system is that a given resource (room, court, desk)
cannot be booked by two users for the same time slot. Under concurrent load,
two requests can pass the "is this slot free?" check at nearly the same instant
and both proceed to insert a booking, producing a double-booking. We need to
guarantee this invariant without serializing all booking traffic, which would
hurt throughput as the number of resources grows.

The application runs against a single PostgreSQL instance and may be scaled to
multiple stateless API instances, so the integrity guarantee must live in the
database, not in application memory.

## Decision

We will enforce booking integrity at two layers:

1. **A database constraint as the source of truth.** A unique/exclusion
   constraint on `(resource_id, time_range)` makes a conflicting insert
   impossible regardless of application logic. For fixed slots this is a plain
   `UNIQUE (resource_id, slot_start)`. For arbitrary time ranges we use a
   PostgreSQL `EXCLUDE` constraint with `btree_gist` to reject overlapping
   ranges.

2. **Optimistic locking with a JPA `@Version` field** on the aggregate so that
   concurrent updates to related state (e.g. a resource's availability summary)
   fail fast with an `OptimisticLockException` rather than silently
   overwriting.

The service layer catches the constraint violation / optimistic-lock failure
and translates it into a `409 Conflict` response, optionally retrying once for
the optimistic-lock case.

## Alternatives considered

- **Pessimistic locking (`SELECT ... FOR UPDATE`).** Simplest to reason about,
  but it holds row locks for the duration of the transaction and serializes
  bookings against the same resource, reducing concurrency. Rejected as the
  default; acceptable only if contention proves to be on individual resources.
- **Application-level lock (e.g. Redis distributed lock).** Adds an external
  dependency and a new failure mode (lock expiry, clock skew) for a problem the
  database can already solve atomically. Overkill for a single-datastore app.
- **No constraint, rely on the availability check.** Rejected — this is the
  bug we are explicitly preventing. The check-then-insert is a classic race
  condition.

## Consequences

- **Positive:** The integrity guarantee is enforced by the database and cannot
  be bypassed by an application bug or a new code path. Concurrency stays high
  because we do not hold locks across the request.
- **Negative:** The service layer must handle constraint violations explicitly
  and map them to a clean API error. Overlapping-range support depends on the
  `btree_gist` extension being enabled.
- **Follow-up:** Document the `409` behavior in the OpenAPI spec so the Angular
  client can show a "slot just got taken" message gracefully.
